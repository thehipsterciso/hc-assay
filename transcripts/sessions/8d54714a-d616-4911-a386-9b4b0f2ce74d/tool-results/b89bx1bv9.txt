### 6. [MEDIUM] cosine_similarity_matrix converts the O(n^2) numpy result to a nested Python list, ~70x-ing peak memory and serializing the win
- **Location:** `src/assay_engine/baseline/primitives.py` 112 (also 115-131)
- **Problem:** _cosine_matrix_numpy computes the n x n similarity matrix vectorized (good), then at line 112 returns `[[float(v) for v in row] for row in m]` — a full nested Python list of n^2 native float objects. A numpy float64 cell is 8 bytes; a boxed Python float is ~24 bytes plus list-pointer overhead (~8 B) and the per-row list headers, i.e. roughly 32-56 B/cell, ~4-7x the dense array. The conversion is also a pure-Python O(n^2) loop that dominates runtime at scale. PoC against the venv (numpy 2.4.6): n=3000, d=384 (MiniLM-size corpus) took 7.76 s and 390 MB of tracked Python allocation; the returned nested list alone is ~576 MB by the cell-overhead estimate. Breaking point: a corpus of ~5-10k embeddings (25M-100M cells) produces a multi-GB nested list and minutes of boxing time on a Mac mini, OOM-risking the box well before the underlying matmul would. The function signature forces the dense materialization regardless of caller need (most callers want top-k neighbors, a threshold graph, or block-wise stats — none need the full boxed matrix).
- **Suggested fix:** Return the numpy ndarray (or a thin read-only view) from the numpy path instead of boxing to nested lists, letting callers index without the 4-7x blow-up; or expose a streaming/blocked API (e.g. top-k per row, or a generator over row blocks) so the full n x n matrix is never materialized.
