harden(p4/lc-robustness): Shared timeout ThreadPoolExecutor leaks a worker on every timeout and is never re...
**Hardening pass 4 finding #19** · severity **low** · layer `langchain` · dimension `lc-robustness`

**Location:** `src/reasoning_client.py:121, 127-149, 152-161`

### Problem
_timeout_pool is a module-global ThreadPoolExecutor(max_workers=8) that is intentionally never shut down. When _with_timeout (line 147) returns via FuturesTimeout, the underlying _run worker is NOT cancelled — ThreadPoolExecutor cannot interrupt a thread blocked in llm.invoke / asyncio.run, so that worker stays blocked on the hung backend call and the slot is gone until (if ever) the call returns. _run_sync (line 161) submits T3 work to the SAME 8-slot pool. On the production 24/7 Mac mini node, each truly-hung Ollama/Agent-SDK call permanently consumes one of 8 slots; after 8 such events the pool is saturated, newly submitted callables never start, and every future _with_timeout/_run_sync call blocks until its own result(timeout) expires — turning every reasoning call into a guaranteed timeout. This is a cumulative, process-lifetime reliability cliff rather than a per-call failure.

### Best practice violated
A bounded global thread pool used to enforce timeouts must not permanently lose capacity to hung work; otherwise accumulated leaks degrade into a hard cliff where every subsequent call is forced to time out.

### Fix
Treat sustained pool saturation as a detectable condition: track in-flight vs. max_workers and surface a distinct ReasoningError (e.g. 'reasoning timeout pool saturated') when no slot is available within a short bound, and/or size the pool relative to expected node concurrency and add a watchdog/metric so leaked workers are visible in Phoenix. ### Adversarial verification
The code-level facts are confirmed: _timeout_pool = ThreadPoolExecutor(max_workers=8) is a module global that is intentionally never shut down (src/reasoning_client.py:118-121), both _with_timeout (line 145) and _run_sync (line 161) submit to this SAME pool, and on FuturesTimeout (line 147-149) the worker is abandoned, not cancelled. CPython's ThreadPoolExecutor genuinely cannot interrupt a thread blocked in a call, so the mechanism the finding describes (leak -> cumulative saturation -> queued callables never start -> every subsequent result(timeout) expires) is mechanically correct. So this is not a misunderstanding of executor semantics, and there is no in-flight tracking, fail-fast on saturation, or watchdog/metric; the design comment only reasons about a SINGLE leaked worker, not cumulative saturation.

However, the finding materially overstates likelihood/severity by asserting that 'each truly-hung call permanently consumes one of 8 slots.' The code has inner per-tier timeouts specifically designed to let the worker self-terminate so the leak is NOT permanent in the common case: T2 sets client_kwargs={'timeout': T2_TIMEOUT} (line 182), an HTTP/socket timeout that makes llm.invoke raise rather than block forever; T3 wraps _drain() in asyncio.wait_for(..., timeout=T3_TIMEOUT) (line 285) so the coroutine self-cancels and asyncio.run returns, and _run_sync's outer backstop is deliberately T3_TIMEOUT + 30 (line 161, comment 158-160) so the inner timeout wins and frees the worker cleanly. A PERMANENT leak therefore requires the inner timeout to ALSO fail (a C-level blocking call ignoring the socket timeout, or a non-cancellable asyncio task) — a narrow degenerate case, not the routine outcome on every backend hiccup. Additionally this is a single-process LangGraph node where node-level reasoning calls are largely sequential, so accumulating 8 concurrent permanent leaks is unusual. A minor real weakness also exists: for T2 the inner HTTP timeout and the outer future timeout are both exactly T2_TIMEOUT, a race where the backstop gives no margin over the inner timeout.

Net: a genuine latent robustness gap (no saturation detection; comment only covers a single leak), but defended in the common case by inner timeouts, so it is lower severity than the claimed 'guaranteed timeout cliff' under normal operation.

### Recommended action
Keep as a low-priority hardening item rather than treating it as an active reliability cliff. Concretely: (1) Add lightweight in-flight tracking against max_workers and surface a distinct ReasoningError ('reasoning timeout pool saturated') when no slot is acquired within a short bound, so cumulative leakage fails loud/fast instead of silently converting all traffic to timeouts. (2) Emit a Phoenix/OTel gauge for active vs. max workers so leaked workers are observable. (3) Give the T2 outer backstop margin over its inner HTTP timeout (e.g. future.result(timeout=T2_TIMEOUT + small_delta)) so the inner timeout reliably wins and frees the worker, mirroring the T3_TIMEOUT + 30 pattern. No urgent change required; the inner per-tier timeouts already prevent permanent leaks in the normal hung-backend case.

### Problem
